# AGENTS.md

本文件为在本仓库中工作的 AI 编码代理提供项目级约定。除非用户明确要求，否则请遵循以下规则。

## 项目概览

- `just-go` 是教材式 Go 学习仓库，不是单一产品应用。
- 项目使用 Go 1.24，根目录只有一个 Go module：`just-go`。
- 内容按学习阶段组织：
  - `stage-1-syntax/`：语法、标准库与基础工程实践。
  - `stage-2-business/`：Web、数据持久化、缓存、消息队列与可观测性。
  - `stage-3-architecture/`：整洁架构、DDD、微服务及综合项目。
  - `docs/`：术语、FAQ、参考资料和设计文档。
  - `openspec/`：规格驱动的提案、设计、规格与任务。
- 每章通常是可独立运行、带测试和练习说明的示例；阶段末尾的 `capstone-*` 是综合项目。

## 修改原则

- 先阅读目标目录的 `README.md`、`EXERCISES.md`、测试及相关 OpenSpec 文档，再修改代码。
- 保持教学代码清晰、渐进且适合当前章节；不要为了抽象或复用引入尚未讲授的复杂模式。
- 优先做范围最小、与需求直接相关的改动，不顺带重构无关章节。
- 保持各章节可独立理解和运行。跨章节复用代码前，先确认不会破坏教学边界。
- 新行为或缺陷修复应配套测试；优先使用表驱动测试和 Go 标准测试工具。
- 不修改生成文件，除非同时修改其来源并重新生成。`stage-3-architecture/12-clean-architecture/wire_gen.go` 由 Wire 生成。
- 不覆盖或清理用户已有的未提交改动。

## Go 代码约定

- 所有 Go 代码必须通过 `gofmt`；遵循惯用 Go 命名和包设计。
- 错误应被处理或显式返回；需要补充语义时使用 `%w` 包装错误。
- 避免全局可变状态、无必要的接口、过早抽象和隐藏控制流。
- 并发代码要明确生命周期、取消机制和资源所有权，并使用 race detector 验证。
- 数据库、文件、HTTP 响应体等资源应及时关闭；相关调用应检查错误。
- 新依赖必须确有必要。添加或删除依赖后运行 `go mod tidy`，并提交同步更新的 `go.mod`、`go.sum`。
- SQLite 相关包使用 `github.com/mattn/go-sqlite3`，运行对应测试或构建时需具备 CGO 和 C 编译器环境。

## 文档与规格

- 面向学习者的说明优先使用简体中文；代码标识符、命令和通用技术名词保持其标准英文形式。
- Markdown、YAML、JSON 使用 2 空格缩进；文件使用 UTF-8、LF，并保留末尾换行。
- 修改章节范围、学习目标或可观察行为时，同步检查对应的 `README.md`、`EXERCISES.md`、`ROADMAP.md` 与 `openspec/` 内容。
- 对已有 OpenSpec change 的实现，以其 `proposal.md`、`design.md`、`specs/` 和 `tasks.md` 为准；完成任务后更新对应复选框。
- 新的大型功能或课程章节优先先创建 OpenSpec proposal；小型修复、文档勘误和不改变行为的整理可直接修改。

## 验证命令

先运行与修改范围最相关的测试，再执行全仓库检查：

```bash
go test ./path/to/changed/package
go test ./...
go vet ./...
go test -race -count=1 ./...
go build ./...
golangci-lint run
```

- CI 使用 Go 1.24 和 golangci-lint v2.1.6。
- 若只修改文档，可不运行 Go 测试，但应检查链接、路径、命令和 Markdown 格式。
- 若环境缺少 CGO、C 编译器、外部服务或 lint 工具，应说明未执行的检查及原因，不要声称验证通过。

## 交付要求

- 最终说明应概括改动、列出实际执行的验证以及仍存在的限制。
- 不提交构建产物、临时文件、编辑器缓存、凭据或本地配置。
- 未经用户明确要求，不创建提交、不推送分支，也不改写 Git 历史。
